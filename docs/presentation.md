## エピソード①：画像 PDF は画像のまま渡さなかった

**状況**

- 旧形式（第 35 回以前）の過去問は**画像 PDF**
- `pdf_to_scaffold_json.py` でまず OCR → テキスト JSON に変換

```python
# 画像 PDF → OCR テキスト → 構造化 JSON の流れ
page_text = extract_text_with_ocr(pdf_page)   # pdfplumber + tesseract
scaffold  = build_question_json(page_text)     # 問題文・選択肢を構造化
```

**Claude に渡すのはこの「構造化 JSON」**

> AI への渡し方にも設計が要る。<br>
> 「とりあえずぶち込む」より「整えてから渡す」が品質に直結した。

<!--
「AI はなんでも読んでくれる」と思いがちだが、画像のまま渡すと解釈ブレが起きやすい。
OCR で一度テキスト化してから渡す設計にした理由と、実際に品質が安定した話をする。
-->

## エピソード②：全問ぶっこんだらコンテキスト爆発した

**やったこと**

全 150 問 × 選択肢 × 正解 × 解説ひな型 → **1 回のプロンプトに全部**

**結果**

| 条件 | 出力の傾向 |
|------|------------|
| 少量（1〜5 問） | 具体的・試験に即した解説 |
| 大量（50 問以上）| 抽象的・同じ文言の繰り返し |

> コンテキストが長くなるほど、AI は「それっぽいが薄い」答えを返してくる。<br>
> 量より、**渡す単位の設計**が大事だと痛感した。

## エピソード③（続き）：実際のやり取りのイメージ

```
[自分]

## 人間が確認する工程をどこに残したか

**AI 生成 → 人間レビュー → DB 投入**

```
generate_explanations.py   # AI で解説を一括生成
        ↓
quiz_editor.html           # ブラウザで 1 問ずつ目視確認・修正
        ↓
import_questions.py        # SQLite に投入
```

**レビュー工程を省けなかった理由**

- AI は「それっぽい解説」を自信満々に書く
- 試験問題は正確性が命。誤解説のまま覚えると本番でアウト

> AI を信頼しつつ、検証は怠らない。<br>
> 自動化と人間の目のバランスが、品質の肝だった。

<!--
quiz_editor.html はブラウザで動くシンプルな編集 UI。
「全部 AI に任せればいい」ではなく、確認工程を設計に組み込んだことを強調する。
-->

## 学んだこと 3 つ

**1. AI への渡し方に設計が要る**
生データをそのまま渡さず、構造化・前処理してから渡す

**2. コンテキストは絞る**
一度に渡す量が多すぎると品質が落ちる。分割が品質管理のカギ

**3. AI は自律的に解決策を出してくることがある**
「相談する」だけでアイデアと実装が返ってくる。問題を伝えることが仕事になる

---

> まとめると：<br>
> AI は「指示した通りに動くツール」というより<br>
> **「問題を一緒に考えるパートナー」に近い。**

3 点は独立しているが、根底にある「AI との協働の設計」というテーマでつながっている。

## 横展開できるパターン & GitHub

**「PDF + AI + 学習ループ」の型**

```
① 生データ（PDF・CSV・Excel）
